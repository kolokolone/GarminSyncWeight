Phase 3 — Récupération des données Withings

Créer un client Withings propre :

```text
backend/app/services/withings_client.py
```

Endpoint Withings à utiliser :

* API Measure / `getmeas`.

Important :

* Vérifie la documentation actuelle Withings avant implémentation finale.
* Ne suppose pas les types de mesures.
* Les types ci-dessous sont des hypothèses documentaires à confirmer dans le code et la documentation.

Types de mesure à gérer prudemment :

* `1` : poids en kg ;
* `5` : fat free mass / masse maigre en kg ;
* `6` : fat ratio / pourcentage de masse grasse ;
* `8` : fat mass weight / masse grasse en kg ;
* `76` : muscle mass / masse musculaire en kg ;
* `77` : hydration / hydratation, probablement en kg selon les réponses ;
* `88` : bone mass / masse osseuse en kg ;
* autres types éventuellement disponibles à documenter mais ne pas mapper sans validation.

Règles de parsing :

* Withings renvoie souvent des mesures sous forme de groupes `measuregrps`.
* Chaque groupe peut contenir plusieurs mesures.
* Chaque mesure peut contenir `type`, `value`, `unit`.
* La valeur réelle doit être reconstruite par :

```python
real_value = value * (10 ** unit)
```

* Le parser doit gérer correctement les unités négatives.
* Le parser doit conserver la réponse brute dans le modèle interne pour audit.
* Les timestamps doivent être traités proprement.
* La timezone doit être explicite, par défaut `Europe/Paris`, configurable via `APP_TIMEZONE`.
* La date Garmin doit être dérivée de façon stable à partir du timestamp local.
* Gérer si disponibles :

  * `grpid`
  * `date`
  * `created`
  * `modified`
  * `attrib`
  * `category`
  * `deviceid`
* Ne pas écraser ou fusionner deux mesures du même jour sans stratégie claire.

Créer un modèle interne canonique.

Exemple :

```python
from datetime import date, datetime
from decimal import Decimal
from typing import Literal, Any
from pydantic import BaseModel

class BodyCompositionMeasurement(BaseModel):
    source: Literal["withings"] = "withings"
    source_measure_group_id: str | None = None
    source_device_id: str | None = None
    measured_at_utc: datetime
    measured_at_local: datetime
    garmin_date: date

    weight_kg: Decimal | None = None
    fat_percent: Decimal | None = None
    fat_mass_kg: Decimal | None = None
    fat_free_mass_kg: Decimal | None = None
    muscle_mass_kg: Decimal | None = None
    bone_mass_kg: Decimal | None = None
    hydration_mass_kg: Decimal | None = None
    hydration_percent: Decimal | None = None
    bmi: Decimal | None = None

    visceral_fat_mass: Decimal | None = None
    basal_met: Decimal | None = None
    active_met: Decimal | None = None
    physique_rating: int | None = None
    metabolic_age: int | None = None

    warnings: list[str] = []
```

Règles :

* Ne pas inventer de données.
* Ne pas calculer de champs Garmin si l’interprétation est douteuse.
* Ne calculer l’IMC que si la taille utilisateur est disponible de manière fiable via configuration explicite `USER_HEIGHT_M`.
* Ne pas calculer `basal_met`, `active_met`, `physique_rating`, `metabolic_age`, `visceral_fat_rating` sans source Withings directe confirmée.
* Ne pas convertir automatiquement `hydration_mass_kg` en `percent_hydration` sauf si la formule est explicitement validée.
* Si hydratation est fournie en kg et Garmin attend un pourcentage, laisser `percent_hydration=null` au départ et signaler dans le rapport.

Logique anti-doublon prudente

Entrées :

* candidat Withings normalisé ;
* weigh-ins Garmin du même jour ;
* body composition Garmin sur une fenêtre autour de la date ;
* historique local `sync_events`.

Fenêtre de comparaison :

* même jour local ;
* éventuellement ±12 h si Garmin renvoie un timestamp ;
* configurable.

Seuils configurables :

* `WEIGHT_DUPLICATE_EPSILON_KG=0.05`
* `WEIGHT_CONFLICT_EPSILON_KG=0.2`
* `GARMIN_LOOKBACK_DAYS=7`
* `GARMIN_LOOKAHEAD_DAYS=1`
* `WITHINGS_PER_DAY_STRATEGY=latest_per_day`

Règles :

1. Si Garmin contient déjà une entrée le même jour avec un poids identique à ±0,05 kg :

   * classer `duplicate_exact_or_near` ;
   * ne pas écrire.
2. Si Garmin contient déjà une entrée le même jour avec une différence de poids ≤0,2 kg :

   * classer `possible_duplicate` ;
   * ne pas écrire par défaut ;
   * signaler doublon probable.
3. Si Garmin contient déjà une composition corporelle le même jour avec poids proche et au moins un champ de composition proche :

   * classer `duplicate_body_composition` ;
   * ne pas écrire.
4. Si Garmin contient une entrée le même jour mais poids différent >0,2 kg :

   * classer `conflict_same_day` ;
   * ne pas écrire ;
   * rapporter explicitement.
5. Si aucune entrée Garmin n’existe le même jour :

   * classer `new_candidate` ;
   * dry-run indique l’appel qui serait fait.
6. Si plusieurs mesures Withings existent le même jour :

   * ne pas toutes écrire automatiquement ;
   * stratégie par défaut : `latest_per_day` ;
   * autres stratégies possibles mais désactivées par défaut :

     * `earliest_per_day` ;
     * `all_if_distinct`.
   * en dry-run, afficher les mesures ignorées.
7. Si poids Withings absent :

   * classer `invalid_missing_weight` ;
   * ne pas écrire.
8. Si date invalide :

   * classer `invalid_date` ;
   * ne pas écrire.
9. Si valeur aberrante :

   * classer `invalid_outlier` ;
   * ne pas écrire.
10. Si une mesure a déjà été importée par notre service :

* vérifier via table locale `sync_events` et `idempotency_key` ;
* classer `already_synced_by_garminsync` ;
* ne pas écrire à nouveau.

Valeurs aberrantes minimales :

* poids < 20 kg ou > 300 kg : invalide ;
* pourcentage de masse grasse < 2 % ou > 75 % : invalide ;
* masse musculaire, osseuse ou hydrique négative : invalide ;
* IMC calculé < 10 ou > 80 : warning ou invalide selon contexte.
