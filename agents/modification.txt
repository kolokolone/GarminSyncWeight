Tu dois auditer puis corriger l’architecture de l’application GarminSyncWeight afin de garantir une séparation nette entre backend et frontend.

Objectif général : le backend doit être la source de vérité pour tous les traitements métier, calculs, synchronisations, validations, mappings, accès API externes, déduplications, décisions de conflit, stockage et génération de rapports. Le frontend doit rester léger, réactif, lisible, et ne doit servir qu’à afficher les données, recueillir les choix utilisateur, déclencher les actions backend et présenter les retours d’état.

Ne pars pas du principe que l’architecture actuelle est correcte. Vérifie réellement le code existant, identifie les responsabilités actuellement portées par le frontend, puis déplace côté backend tout ce qui relève du métier.

1. Audit frontend/backend

Analyse tout le dépôt GarminSyncWeight et produis d’abord un état des lieux clair :

* quelles fonctions sont exécutées côté backend ;
* quelles fonctions sont exécutées côté frontend ;
* quels appels API sont faits par le frontend ;
* quels calculs sont faits dans le navigateur ;
* quelles responsabilités métier sont mal placées ;
* quelles routes backend existent déjà et peuvent être réutilisées ;
* quelles routes backend doivent être améliorées ou créées ;
* quels fichiers frontend sont des sources modifiables et quels fichiers sont des fichiers générés/buildés.

Point important : ne modifie pas directement un bundle généré comme `frontend/out/assets/app.js` si des sources frontend existent ailleurs. Corrige les vrais fichiers sources, puis reconstruis proprement. Si le projet ne contient réellement que des fichiers statiques générés, signale-le clairement dans le rapport et applique une correction minimale propre, sans masquer le problème de maintenabilité.

2. Règle d’architecture à appliquer

Le frontend peut :

* afficher les mesures Withings et Garmin ;
* afficher les statuts de connexion ;
* afficher les erreurs et les messages de succès ;
* proposer des boutons d’action ;
* recueillir email, mot de passe, OTP, choix de période, paramètres d’affichage ;
* appeler les endpoints backend ;
* écouter les événements SSE ou WebSocket si utilisés ;
* afficher des graphiques, tableaux, badges et états de chargement ;
* gérer uniquement l’état UI local : onglet actif, formulaire ouvert, loader, message visible, ligne sélectionnée.

Le frontend ne doit pas :

* calculer l’IMC ;
* mapper les champs Withings vers Garmin ;
* décider si une mesure doit être synchronisée ;
* dédupliquer les mesures ;
* décider si une mesure Garmin existe déjà ;
* interpréter les formats Garmin ou Withings ;
* appliquer les règles métier de synchronisation ;
* calculer les périodes finales en source de vérité ;
* corriger les dates, timezone ou bornes temporelles ;
* générer des rapports métier ;
* faire des appels directs à Garmin ou Withings ;
* contenir des secrets, tokens, mots de passe persistés ou logique d’authentification sensible.

Le backend doit :

* gérer l’authentification Withings ;
* gérer l’authentification Garmin ;
* gérer le flux OTP Garmin ;
* gérer les tokens et sessions ;
* récupérer les données Withings ;
* récupérer les données Garmin ;
* parser les mesures ;
* normaliser les dates ;
* appliquer la timezone ;
* calculer l’IMC ;
* mapper les mesures Withings vers Garmin ;
* comparer les mesures Withings et Garmin ;
* détecter les doublons ;
* décider ce qui est synchronisable ;
* écrire dans Garmin ;
* stocker l’historique local ;
* construire les rapports ;
* exposer au frontend des objets déjà prêts à afficher.

3. Correction du calcul d’IMC

Cherche tout calcul d’IMC fait côté frontend, notamment les formules du type :

poids / taille²

ou toute variante utilisant `weight_kg`, `height_cm`, `height_m`, `bmi`.

Déplace ce calcul côté backend.

Le backend doit exposer une valeur d’IMC déjà calculée dans la réponse API, avec un format explicite. Exemple attendu :

{
"metric": "bmi",
"value": 24.5,
"source": "computed_backend",
"inputs": {
"weight_kg": 92.1,
"height_cm": 194
}
}

Le frontend ne doit plus refaire le calcul. Il doit simplement afficher la valeur reçue.

Si Withings fournit déjà un IMC, le backend doit gérer la priorité clairement :

* utiliser la valeur Withings si elle existe et paraît valide ;
* sinon calculer l’IMC côté backend à partir du poids et de la taille configurée ;
* indiquer dans la réponse si la valeur est fournie ou calculée.

Ajouter des tests backend pour :

* poids valide + taille valide ;
* taille absente ;
* taille invalide ;
* poids absent ;
* arrondi ;
* comparaison entre IMC fourni par Withings et IMC calculé localement.

4. Correction de la gestion des périodes de synchronisation

Le frontend peut proposer des choix simples :

* dernière mesure ;
* aujourd’hui ;
* 7 jours ;
* 30 jours ;
* période personnalisée.

Mais le backend doit rester la source de vérité.

Le frontend peut envoyer une intention, par exemple :

{
"period": "7d"
}

ou :

{
"start_date": "2026-06-01",
"end_date": "2026-06-28"
}

Mais le backend doit :

* valider la période ;
* normaliser les dates ;
* appliquer la timezone configurée ;
* refuser les périodes inversées ;
* refuser les périodes trop longues si nécessaire ;
* refuser les dates futures incohérentes ;
* documenter la période réellement utilisée dans la réponse.

La réponse backend doit renvoyer les bornes réellement prises en compte, par exemple :

{
"requested_period": "7d",
"resolved_start": "2026-06-21T00:00:00+02:00",
"resolved_end": "2026-06-28T23:59:59+02:00",
"timezone": "Europe/Paris"
}

Le frontend doit afficher ces informations si utile, mais ne doit pas les recalculer.

5. Correction du flux Garmin OTP

Le problème utilisateur actuel est le suivant : après avoir cliqué sur “Connecter Garmin”, Garmin demande un code OTP. L’utilisateur saisit le code, mais l’interface ne rend pas évident le bouton à cliquer pour valider le code et finaliser la connexion.

Tu dois corriger ce flux de manière explicite.

Comportement attendu :

Étape 1 : formulaire Garmin initial.

Champs :

* email Garmin ;
* mot de passe Garmin ;
* bouton principal : “Connecter Garmin”.

Au clic, le frontend appelle :

POST /api/garmin/auth/login

avec email et mot de passe.

Étape 2 : si le backend répond que l’OTP est nécessaire.

Si la réponse contient une information du type :

{
"needs_otp": true
}

alors le frontend doit afficher immédiatement un bloc OTP clair.

Ce bloc doit contenir :

* un message du type : “Garmin demande un code de vérification. Saisis le code reçu, puis valide la connexion.”
* un champ “Code OTP Garmin” ;
* un seul bouton principal : “Valider le code OTP et connecter Garmin”.

Ce bouton doit appeler à nouveau :

POST /api/garmin/auth/login

avec :

* email ;
* mot de passe ou identifiant de session temporaire si le backend est modifié pour le permettre ;
* otp.

Il ne faut pas utiliser `/api/garmin/auth/verify` pour soumettre l’OTP.

La route `/api/garmin/auth/verify` doit rester une route de vérification d’état uniquement. Elle peut servir à vérifier si une connexion Garmin déjà existante est valide, mais pas à valider un code OTP.

6. Amélioration backend du flux OTP

Vérifie l’implémentation actuelle de l’authentification Garmin.

Si le backend relance entièrement le processus de login après réception de l’OTP, évalue si c’est fragile. Si possible, améliore le flux avec une session temporaire.

Comportement idéal :

Premier appel :

POST /api/garmin/auth/login

Réponse :

{
"success": false,
"needs_otp": true,
"auth_session_id": "temporary-session-id",
"message": "OTP required"
}

Second appel :

POST /api/garmin/auth/login

Payload :

{
"auth_session_id": "temporary-session-id",
"otp": "123456"
}

Le backend doit alors reprendre le flux d’authentification Garmin associé à cette session temporaire.

La session temporaire doit :

* expirer rapidement ;
* ne pas exposer le mot de passe au frontend après le premier envoi ;
* ne jamais être persistée durablement ;
* ne jamais être loggée avec des données sensibles ;
* être supprimée après succès ou échec définitif.

Si cette amélioration est trop lourde pour l’architecture actuelle, applique une version minimale acceptable :

* le second appel `/login` accepte email + mot de passe + otp ;
* le frontend garde temporairement les valeurs uniquement en mémoire pendant le flux ;
* l’interface explique clairement quoi faire ;
* les erreurs OTP sont compréhensibles ;
* les champs sensibles ne sont jamais stockés durablement ;
* les logs ne contiennent jamais le mot de passe ni l’OTP.

7. Messages d’erreur Garmin à améliorer

Les erreurs doivent être explicites côté utilisateur.

Prévoir au minimum les cas suivants :

* identifiants Garmin incorrects ;
* OTP requis ;
* OTP invalide ;
* OTP expiré ;
* délai dépassé ;
* Garmin indisponible ;
* connexion réussie ;
* token Garmin déjà valide ;
* déconnexion réussie ;
* échec de vérification du statut Garmin.

Ne pas afficher de stack trace brute dans l’interface.

Côté logs backend, conserver des logs utiles techniquement, mais sans secret :

* ne jamais logger le mot de passe ;
* ne jamais logger l’OTP ;
* ne jamais logger les tokens complets ;
* masquer les emails si nécessaire ;
* garder les étapes : login started, otp required, otp submitted, login success, login failed.

8. Clarification des boutons Garmin dans l’interface

La page Garmin doit avoir une hiérarchie claire.

Boutons attendus :

* “Connecter Garmin” : démarre la connexion avec email/mot de passe.
* “Valider le code OTP et connecter Garmin” : visible uniquement quand Garmin demande un OTP.
* “Vérifier la connexion Garmin” : vérifie seulement si la connexion actuelle est valide.
* “Déconnecter Garmin” : supprime ou invalide la session/token local selon l’implémentation.

Le bouton “Vérifier la connexion Garmin” ne doit jamais être placé ou formulé de manière à laisser croire qu’il valide l’OTP.

Après succès OTP, le frontend doit :

* masquer le bloc OTP ;
* afficher “Garmin connecté” ;
* rafraîchir le statut Garmin ;
* permettre de lancer une synchronisation ;
* ne pas laisser l’utilisateur bloqué sur le formulaire OTP.

9. Allègement du chargement frontend

Auditer les appels faits au chargement initial de l’application.

Si `/api/status` déclenche des vérifications externes lentes Garmin ou Withings, corriger.

Comportement attendu :

* `/api/status` doit être rapide ;
* il doit retourner l’état local connu ;
* il ne doit pas bloquer l’interface si Garmin ou Withings est lent ;
* les vérifications externes doivent être déclenchées séparément ;
* le frontend doit afficher la page immédiatement ;
* les statuts externes doivent être chargés progressivement.

Exemple de séparation souhaitée :

GET /api/status
Retourne l’état local de l’application.

GET /api/garmin/auth/status
Retourne le statut Garmin local ou vérifié selon l’implémentation actuelle.

POST /api/garmin/auth/verify
Lance une vérification explicite de la connexion Garmin existante.

GET /api/withings/auth/status
Retourne le statut Withings.

Le frontend ne doit pas attendre tous les statuts pour rendre la page.

12. Gestion des événements de progression

Si l’application utilise SSE pour suivre la synchronisation, conserver ce principe, mais vérifier que le frontend n’interprète pas le métier.

Le backend doit envoyer des événements déjà qualifiés :

* `sync_started`;
* `withings_fetch_started`;
* `withings_fetch_completed`;
* `garmin_fetch_started`;
* `garmin_fetch_completed`;
* `deduplication_completed`;
* `sync_write_started`;
* `sync_write_completed`;
* `sync_completed`;
* `sync_failed`.

Chaque événement doit contenir un message prêt à afficher et éventuellement des compteurs.

Le frontend ne doit pas reconstruire la logique métier à partir d’événements trop bruts.
